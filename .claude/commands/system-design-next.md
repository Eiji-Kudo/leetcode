# システムデザイン 次のステップを進める

`docs/system-design-plan.md` の進捗を確認し、次の未完了ステップのテンプレートを生成してユーザーの学習を進める。
復習スケジュール（間隔反復）も管理し、適切なタイミングで復習を促す。

## 手順

### 0. 復習スケジュールの確認

`docs/system-design-progress.json` を読む。存在しない場合は `{"completions": {}}` で初期化してファイルを作成する。

本日の日付と各ステップの `nextReview` を比較し、**復習期日が今日以前のステップ**を抽出する。

#### system-design-progress.json の構造

```json
{
  "completions": {
    "1-1": {
      "completedAt": "2026-03-07",
      "reviews": [
        { "date": "2026-03-08", "passed": true },
        { "date": "2026-03-11", "passed": true }
      ],
      "intervalLevel": 2,
      "nextReview": "2026-03-18"
    }
  }
}
```

#### 間隔反復アルゴリズム（Spaced Repetition）

エビングハウスの忘却曲線に基づく復習間隔:

| intervalLevel | 次の復習まで | 意味 |
|---|---|---|
| 0 | 1日後 | 翌日の即時復習 |
| 1 | 3日後 | 短期記憶の定着 |
| 2 | 7日後 | 1週間後の確認 |
| 3 | 14日後 | 2週間後の確認 |
| 4 | 30日後 | 1ヶ月後の最終確認 |
| 5 | なし（卒業） | 長期記憶に定着済み |

**成功時**: `intervalLevel` を +1 し、対応する日数後を `nextReview` に設定する。Level 5 で卒業（`nextReview: null`）。

**失敗時**: `intervalLevel` を `max(0, current - 1)` に戻し、対応する日数後を `nextReview` に再設定する。

### 1. 進捗の確認

`docs/system-design-plan.md` を読み、チェックボックスの状態を確認する。

- `- [x]` → 完了済み
- `- [ ]` → 未完了

最初の未完了ステップ（`- [ ]`）を「次のステップ」として特定する。

完了済みステップの総数と全体数を数え、進捗サマリーを表示する。

**復習が必要なステップがある場合:**

```
📊 進捗: 5/83 完了（Step 1: ネットワーク・Web基礎 - 1-5 まで完了）
[███░░░░░░░░░░░░░░░░░] 6%
🔔 復習が必要: 1-3「TCP と UDP」(7日前に完了、復習期日: 3日前)
📅 全完了予測: 2026年6月10日（残り78ステップ / 1.0ステップ/日ペース）
🎓 卒業予測: 2026年8月4日（完了+復習卒業55日）
👉 まず復習から始めましょう → 1-3 の復習問題を出題します
```

復習が複数ある場合は**期日が最も古いもの**（最も遅延しているもの）を優先する。

**復習がない場合:**

```
📊 進捗: 5/83 完了（Step 1: ネットワーク・Web基礎 - 1-5 まで完了）
[███░░░░░░░░░░░░░░░░░] 6%
✅ 復習: 全て最新（次の復習: 1-3 → 3月15日）
📅 全完了予測: 2026年6月10日（残り78ステップ / 1.0ステップ/日ペース）
🎓 卒業予測: 2026年8月4日（完了+復習卒業55日）
👉 次のステップ: 1-6 WebSocket・ポーリング・SSE
```

**プログレスバーの描画ルール:**
- 全体幅20文字で表示する
- 完了割合に応じて `█`（塗り）と `░`（未塗り）を使い分ける
- `█` の数 = `Math.round(完了数 / 83 * 20)`
- パーセンテージは小数点以下なしで表示する（例: 6%）

### 2. ステップの種類を判断

**復習期日のステップがある場合 → 間隔反復復習へ**（手順2-Rへ）

**復習がない場合**、次のステップのIDと説明から種類を判断する:

- **概念学習ステップ**（例: 1-1, 2-1, 5-1 など）→ **レッスンフェーズへ**（手順2-Lへ）
- **練習ステップ**（例: 1-7, 2-9 など `練習:` がついたもの）→ テンプレート生成へ（レッスン不要、直接設計課題を出す）
- **復習ステップ**（🔁マーク付き）→ 復習テンプレート生成へ（レッスン不要）
- **設計問題の計画ステップ**（8-1a, 9-1a など `a` 付き）→ 要件定義 + 見積もりテンプレートへ
- **設計問題の実装ステップ**（8-1b, 9-1b など `b` 付き）→ 設計テンプレートへ

#### 手順 2-L: レッスンフェーズ（概念学習ステップ）

概念学習ステップでは、**まずレッスン（授業）を行い、理解を確認してから練習に進む**。
ファイルの生成はまだ行わない。

**フェーズの判定方法:**
- 対応するステップのファイルがまだ存在しない → レッスンフェーズ
- ファイルが既に存在し TODO が残っている → ユーザーが演習中（レビュー待ち）
- ユーザーが「わかった」「把握した」「OK」「次へ」等と言った → 次のパートへ進むか、練習フェーズへ移行（手順3へ）

**レッスンの進め方: 少しずつ・対話的に**

一度に全部を説明するのではなく、**2〜3パートに分けて少しずつ教える**。
各パートの後に理解度を確認し、ユーザーの反応を見てから次に進む。

**パート構成（概念の内容に応じて2〜3パートに分ける）:**

**パート1: 概念の導入**
- その概念が何か、なぜシステムデザインで重要か（2〜3文で簡潔に）
- 基本的な仕組みを図解（Mermaid）
- 最もシンプルな例を1つ
- 「ここまで大丈夫ですか？」と確認

**パート2: 実践的な例**
- 実際のサービスでの適用例を1〜2個
- 段階的に複雑さを上げる
- 「ここまで大丈夫ですか？」と確認

**パート3（必要な場合のみ）: 比較・応用**
- 似た概念との比較や使い分け
- よくあるアンチパターンや注意点
- 「OKなら練習に進みましょう！」と案内

**各パートの分量の目安:**
- 1パートにつき図（Mermaid）は1つまで（Claudeが説明用に描く。ユーザーには描かせない）
- 説明テキストは3〜5文まで
- 読むのに1分以内で済む量

**ユーザーの反応に応じた対応:**
- 質問があれば答える
- 「わかった」「OK」等 → 次のパートへ進む（最後のパートなら手順3へ）
- 「もっと詳しく」等 → 同じパートを掘り下げる

**パート1の出力フォーマット例:**

```
## 📖 レッスン: ロードバランサ（1/3）

### ロードバランサとは？
ロードバランサは、複数のサーバーにトラフィックを分散させる装置（またはソフトウェア）です。
1台のサーバーに負荷が集中するのを防ぎ、システムの可用性とスケーラビリティを向上させます。

### 基本的な仕組み

\```mermaid
graph LR
    Client --> LB[Load Balancer]
    LB --> Server1
    LB --> Server2
    LB --> Server3
\```

クライアントは直接サーバーにアクセスせず、ロードバランサを経由します。
ロードバランサが適切なサーバーにリクエストを振り分けます。

---
ここまで大丈夫ですか？
```

**Mermaid図のルール:**
- 色は使わない（`style` や `classDef` で色を付けない）
- シンプルで読みやすい図にする

#### 手順 2-R: 間隔反復復習の出題

復習対象ステップの元ファイルを読み、そのステップで学んだ概念を把握する。
**元の問題と同じ問題は出さず**、同じ概念を使う**新しい問題**を1問出題する。

ファイル名: `review-sr-{ステップID}.md`（例: `review-sr-1-3.md`）
配置先: 元のステップと同じディレクトリ

テンプレート:

```markdown
<!-- =============================================
  間隔反復復習: Step {ID}「{テーマ}」
  完了日: {completedAt} / 復習回数: {reviews.length}回目
============================================= -->

## 元のステップで学んだ概念:
- {概念の要約}

## 復習課題: {新しい問題の説明}

### あなたの回答

<!-- TODO: ここに回答を書いてください -->
```

復習課題の種類（ステップの内容に応じて選ぶ）:
- **概念説明**: その概念を自分の言葉で説明する
- **比較問題**: 2つの技術やアプローチを比較する
- **設計判断**: 特定のシナリオでどの技術を選ぶか理由付きで回答する
- **構成説明**: システムの構成要素とデータの流れをテキストで説明する

### 3. ディレクトリの準備

対応するステップのディレクトリが存在しない場合は作成する:
- Step 1 → `src/sd-step01-network/`
- Step 2 → `src/sd-step02-database/`
- Step 3 → `src/sd-step03-cache/`
- Step 4 → `src/sd-step04-async/`
- Step 5 → `src/sd-step05-distributed/`
- Step 6 → `src/sd-step06-estimation-api/`
- Step 7 → `src/sd-step07-interview/`
- Step 8 → `src/sd-step08-basic-design/`
- Step 9 → `src/sd-step09-intermediate-design/`
- Step 10 → `src/sd-step10-advanced-design/`

### 4. テンプレートファイルの生成

plan.md のステップ情報（ID、テーマ、内容、ファイル名）に基づいて、以下の構成でテンプレートファイルを生成する。

**重要: 1ファイル = 1課題の原則**

1つのステップで複数の概念を扱う場合でも、**1ファイルにつき1つの課題（1つのTODO）** にする。
ユーザーが1つずつ考える→記述→レビューできるようにするため。

**概念学習ステップのテンプレート（レッスン後）:**

レッスンフェーズで概念の説明は済んでいるので、テンプレートには説明を入れない。
課題だけのシンプルな構成にする。

```markdown
<!-- =============================================
  Step {ID}: {テーマ}
============================================= -->

## 課題: {問題の説明}

### あなたの回答

<!-- TODO: ここに回答を書いてください -->
```

課題の種類は概念に応じて選ぶ:
- 概念を自分の言葉で説明する
- 特定シナリオでの適用を考える
- システム構成をテキストで説明する（図はレビュー時にClaudeが描く）
- 2つの技術を比較する

**練習ステップのテンプレート:**

練習ステップはレッスンなしで直接課題を出す。
ただし、課題の背景と条件は十分に記載する。

```markdown
<!-- =============================================
  Step {ID}: {テーマ}
============================================= -->

## 設計課題: {問題の説明}

### 背景・条件
- {条件1}
- {条件2}

### あなたの回答

#### 要件の整理
<!-- TODO: 機能要件と非機能要件を整理してください -->

#### アーキテクチャ・コンポーネント
<!-- TODO: システムの構成要素と、データがどう流れるかを説明してください -->
<!-- 例: クライアント → ロードバランサ → APIサーバー → DB のように -->

#### トレードオフ・考慮点
<!-- TODO: 設計のトレードオフを記述してください -->
```

**復習ステップのテンプレート:**
- 復習対象のステップ範囲から混合問題を出題
- 過去のステップで学んだ概念を組み合わせた設計課題にする
- どの概念を使うかのヒントは入れない（自分で判断させる）

**設計問題 計画ステップ（a付き）のテンプレート:**

```markdown
<!-- =============================================
  Step {ID}: {問題名} — 要件定義 & 見積もり
============================================= -->

## 問題: {問題の概要説明}

### 機能要件
<!-- TODO: 主要な機能を列挙してください（5〜8個） -->

### 非機能要件
<!-- TODO: スケール、レイテンシ、可用性、一貫性等を定義してください -->

### 見積もり
<!-- TODO: 以下を計算してください -->
<!-- - DAU (Daily Active Users) -->
<!-- - Read/Write QPS -->
<!-- - ストレージ（1日/1年/5年） -->
<!-- - 帯域 -->

### 使いそうなコンポーネント・技術
<!-- TODO: この設計に必要そうな技術要素を列挙してください -->
```

**設計問題 実装ステップ（b付き）のテンプレート:**

```markdown
<!-- =============================================
  Step {ID}: {問題名} — 設計 & Deep Dive
============================================= -->

## High-Level Design

### アーキテクチャ
<!-- TODO: システムの構成要素と、データの流れをテキストで説明してください -->
<!-- 例: クライアント → CDN → ロードバランサ → APIサーバー群 → DB -->
<!-- 各コンポーネントの役割も簡潔に書いてください -->

### API設計
<!-- TODO: 主要なAPIエンドポイントを定義してください -->
<!-- 例: POST /api/v1/resource -->

### データモデル
<!-- TODO: 主要なテーブル/コレクションのスキーマを設計してください -->

## Deep Dive

### スケーリング戦略
<!-- TODO: ボトルネックとスケーリング方法を説明してください -->

### 障害対策
<!-- TODO: 起こりうる障害シナリオと対策を記述してください -->

### トレードオフ
<!-- TODO: 設計判断のトレードオフを説明してください -->
```

### 5. ユーザーへの案内

**レッスンフェーズの場合（手順2-Lで概念説明をした直後）:**
- ファイルは生成しない
- 「ここまでの内容は大丈夫ですか？質問があれば聞いてください。OKなら練習に進みましょう！」と案内する

**演習テンプレートを生成した場合:**
1. テンプレートファイルのパス
2. 「TODO部分を記述したら教えてください。レビューします！」

### 6. レビュー（ユーザーが記述後に再度このスキルを呼んだ場合）

ユーザーが「できた」「レビューして」等と言った場合:

1. 該当ファイルを読む
2. 記述内容を以下の観点で評価:
   - **完成度**: 全セクションが埋まっているか
   - **正確性**: 概念の理解や技術の適用が正しいか
   - **深さ**: トレードオフや代替案が考慮されているか
   - **明確さ**: 説明が分かりやすいか、構成が伝わるか
3. 評価結果に応じてフィードバック:
   - **重大な誤りや欠落がある場合** → 具体的に指摘する（答えはユーザーが求めるまで出さない）
   - **概ね良好** → **ディスカッションフェーズ**（手順6-D）へ

#### 手順 6-D: ディスカッションフェーズ

レビュー通過後、すぐに完了にせず、設計についてディスカッションする。
以下の観点でユーザーの設計をレビューし、対話を促す:

1. **設計の改善点**（あれば）
   - 見落としているコンポーネントや考慮事項
   - より適切なパターンや技術選択
   - スケーラビリティやパフォーマンスの観点

2. **別のアプローチ**
   - 同じ問題を別のアーキテクチャで解く方法
   - 実際の企業がどう解決しているかの事例
   - なぜそのアプローチが有効か、どんな規模で使い分けるか

3. **発展的な問い**（任意）
   - 「もしユーザー数が10倍になったら？」
   - 「この部分が障害を起こしたらどうなる？」
   - 「グローバル展開する場合は何が変わる？」

**フォーマット例:**

```
✅ レビュー通過！

### フィードバック
（改善点や補足があればここに。なければ「要点を押さえた良い設計です」等）

### 別のアプローチ
（別のアーキテクチャや技術選択を1つ紹介）

### 実サービスでの事例
（関連する実際のサービスの設計事例を紹介）

---
質問や気になることはありますか？
OKなら次のステップに進みましょう！
```

4. ユーザーが質問したら答える。ディスカッションを続ける。
5. ユーザーが「OK」「次へ」等と言ったら、ステップを完了にする:
   - `system-design-plan.md` のチェックボックスを `- [x]` に更新
   - **system-design-progress.json を更新する**（→ 手順7へ）
   - 「次のステップに進むには `/system-design-next` と入力してください」と案内する

### 7. system-design-progress.json の更新

#### 通常ステップ完了時

レビュー通過後、`docs/system-design-progress.json` に以下を記録する:

```json
{
  "ステップID": {
    "completedAt": "YYYY-MM-DD（本日の日付）",
    "reviews": [],
    "intervalLevel": 0,
    "nextReview": "YYYY-MM-DD（明日の日付）"
  }
}
```

#### 間隔反復復習の完了時

**レビュー通過（成功）の場合:**

1. `reviews` 配列に `{ "date": "本日", "passed": true }` を追加
2. `intervalLevel` を +1
3. 新しい `intervalLevel` に応じて `nextReview` を計算:
   - Level 1 → 本日 + 3日
   - Level 2 → 本日 + 7日
   - Level 3 → 本日 + 14日
   - Level 4 → 本日 + 30日
   - Level 5 → `null`（卒業）
4. 復習用ファイル（`review-sr-*.md`）は削除せず残す

**レビュー不合格の場合:**

1. `reviews` 配列に `{ "date": "本日", "passed": false }` を追加
2. `intervalLevel` を `max(0, current - 1)` に戻す
3. 新しい `intervalLevel` に応じて `nextReview` を再計算
4. 具体的な改善点を示して再挑戦を促す

#### plan.md の🔁復習ステップ完了時

通常のplan.md復習ステップ（🔁マーク付き）は、間隔反復とは別の仕組み。
チェックボックスの更新のみ行い、system-design-progress.json には記録しない。

### 8. 復習状況のサマリー表示

進捗表示時に、復習状況もあわせて表示する:

```
📊 進捗: 10/83 完了
[██████░░░░░░░░░░░░░░] 12%
🧠 復習状況:
  - 卒業済み: 1-1, 1-2, 1-3（3ステップ）
  - 次の復習: 1-4 → 3月12日, 1-5 → 3月15日
  - 復習期日超過: なし
📅 全完了予測: 2026年6月15日（残り73ステップ / 1.0ステップ/日ペース）
🎓 卒業予測: 2026年8月9日（完了+復習卒業55日）
👉 次のステップ: 2-1 ...
```

### 9. 予測日の表示

進捗サマリーに2つの予測を表示する。

#### A. 全完了予測（📅）

全83ステップを解き終わる予測日。

**計算ロジック:**

1. `system-design-progress.json` の全ステップの `completedAt` から、最も古い日付（学習開始日）を取得する
2. 完了ステップ数を数える（progress.json のキー数）
3. 学習開始日から本日までの経過日数を計算する（開始日を含むので `本日 - 開始日 + 1`）
4. ペース = 完了ステップ数 ÷ 経過日数（ステップ/日）
5. 残りステップ数 = 83 - 完了ステップ数
6. 残り日数 = 残りステップ数 ÷ ペース（小数点以下切り上げ）
7. 全完了予測日 = 本日 + 残り日数

#### B. 卒業予測（🎓）

全83ステップの間隔反復が卒業（intervalLevel 5）に到達する予測日。

**計算ロジック:**

最後に完了するステップが卒業するまでに、intervalLevel 0→5 の全復習間隔を経る必要がある:
- Level 0→1: 1日後
- Level 1→2: 3日後
- Level 2→3: 7日後
- Level 3→4: 14日後
- Level 4→5: 30日後
- 合計: **55日**

卒業予測日 = 全完了予測日 + 55日

#### 表示フォーマット

```
📅 全完了予測: YYYY年M月D日（残りNステップ / X.Xステップ/日ペース）
🎓 卒業予測: YYYY年M月D日（完了+復習卒業55日）
```

- ペースは小数第1位まで表示する
- 完了ステップが0の場合（初回起動時）は予測を表示しない
- 学習開始日と本日が同日の場合は、その日のペースで計算する（経過日数 = 1）
