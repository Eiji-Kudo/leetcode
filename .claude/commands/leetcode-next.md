# LeetCode 次のステップを進める

`docs/plan.md` の進捗を確認し、次の未完了ステップのテンプレートを生成してユーザーの学習を進める。
復習スケジュール（間隔反復）も管理し、適切なタイミングで復習を促す。

## 手順

### 0. 復習スケジュールの確認

`docs/progress.json` を読む。存在しない場合は `{"completions": {}}` で初期化してファイルを作成する。

本日の日付と各ステップの `nextReview` を比較し、**復習期日が今日以前のステップ**を抽出する。

#### progress.json の構造

```json
{
  "completions": {
    "1-1": {
      "completedAt": "2026-03-07",
      "reviews": [
        { "date": "2026-03-08", "passed": true },
        { "date": "2026-03-11", "passed": true }
      ],
      "intervalLevel": 2,
      "nextReview": "2026-03-18"
    }
  }
}
```

#### 間隔反復アルゴリズム（Spaced Repetition）

エビングハウスの忘却曲線に基づく復習間隔:

| intervalLevel | 次の復習まで | 意味 |
|---|---|---|
| 0 | 1日後 | 翌日の即時復習 |
| 1 | 3日後 | 短期記憶の定着 |
| 2 | 7日後 | 1週間後の確認 |
| 3 | 14日後 | 2週間後の確認 |
| 4 | 30日後 | 1ヶ月後の最終確認 |
| 5 | なし（卒業） | 長期記憶に定着済み |

**成功時**: `intervalLevel` を +1 し、対応する日数後を `nextReview` に設定する。Level 5 で卒業（`nextReview: null`）。

**失敗時**: `intervalLevel` を `max(0, current - 1)` に戻し、対応する日数後を `nextReview` に再設定する。

### 1. 進捗の確認

`docs/plan.md` を読み、チェックボックスの状態を確認する。

- `- [x]` → 完了済み
- `- [ ]` → 未完了

最初の未完了ステップ（`- [ ]`）を「次のステップ」として特定する。

完了済みステップの総数と全体数を数え、進捗サマリーを表示する。

**復習が必要なステップがある場合:**

```
📊 進捗: 5/82 完了（Step 1: 配列の基本操作 - 1-6 まで完了）
[███░░░░░░░░░░░░░░░░░] 6%
🔔 復習が必要: 1-3「for ループで配列を走査」(7日前に完了、復習期日: 3日前)
📅 Phase1-2完了予測: 2026年6月5日（残り77ステップ / 1.0ステップ/日ペース）
🎓 Phase1-2卒業予測: 2026年7月30日（完了+復習卒業55日）
🏆 Medium安定予測: 2026年12月22日（全~220ステップ+復習卒業55日）
👉 まず復習から始めましょう → 1-3 の復習問題を出題します
```

復習が複数ある場合は**期日が最も古いもの**（最も遅延しているもの）を優先する。

**復習がない場合:**

```
📊 進捗: 5/82 完了（Step 1: 配列の基本操作 - 1-6 まで完了）
[███░░░░░░░░░░░░░░░░░] 6%
✅ 復習: 全て最新（次の復習: 1-3 → 3月15日）
📅 Phase1-2完了予測: 2026年6月5日（残り77ステップ / 1.0ステップ/日ペース）
🎓 Phase1-2卒業予測: 2026年7月30日（完了+復習卒業55日）
🏆 Medium安定予測: 2026年12月22日（全~220ステップ+復習卒業55日）
👉 次のステップ: 1-7 reduce — 各要素を変換して新しい配列を返す
```

**プログレスバーの描画ルール:**
- 全体幅20文字で表示する
- 完了割合に応じて `█`（塗り）と `░`（未塗り）を使い分ける
- `█` の数 = `Math.round(完了数 / 82 * 20)`
- パーセンテージは小数点以下なしで表示する（例: 6%）

### 2. ステップの種類を判断

**復習期日のステップがある場合 → 間隔反復復習へ**（手順2-Rへ）

**復習がない場合**、次のステップのIDと説明から種類を判断する:

- **概念学習ステップ**（例: 1-1, 2-1, 5-1 など）→ **レッスンフェーズへ**（手順2-Lへ）
- **練習ステップ**（例: 1-9, 2-10 など `練習:` がついたもの）→ テンプレート生成へ（レッスン不要、直接問題を出す）
- **復習ステップ**（🔁マーク付き）→ 復習テンプレート生成へ（レッスン不要）
- **LeetCode問題の計画ステップ**（7-1a, 8-1a など `a` 付き）→ 問題解説 + アプローチ検討テンプレートへ
- **LeetCode問題の実装ステップ**（7-1b, 8-1b など `b` 付き）→ 実装テンプレートへ

#### 手順 2-L: レッスンフェーズ（概念学習ステップ）

概念学習ステップでは、**まずレッスン（授業）を行い、理解を確認してから練習問題に進む**。
ファイルの生成はまだ行わない。

**フェーズの判定方法:**
- 対応するステップのファイルがまだ存在しない → レッスンフェーズ
- ファイルが既に存在し TODO が残っている → ユーザーが練習中（レビュー待ち）
- ユーザーが「わかった」「把握した」「OK」「次へ」等と言った → 次のパートへ進むか、練習フェーズへ移行（手順3へ）

**レッスンの進め方: 少しずつ・対話的に**

一度に全部を説明するのではなく、**2〜3パートに分けて少しずつ教える**。
各パートの後に理解度を確認し、ユーザーの反応を見てから次に進む。

**パート構成（概念の内容に応じて2〜3パートに分ける）:**

**パート1: 概念の導入**
- その概念が何か、なぜ重要か（2〜3文で簡潔に）
- 基本構文を提示
- 最もシンプルな例を1つ
- 「ここまで大丈夫ですか？」と確認

**パート2: 実践的な例**
- もう少し実用的な例を1〜2個
- 段階的に複雑さを上げる
- 「ここまで大丈夫ですか？」と確認

**パート3（必要な場合のみ）: 比較・応用**
- 似た概念との比較や使い分け
- よくある間違いや注意点
- 「OKなら練習問題に進みましょう！」と案内

**各パートの分量の目安:**
- 1パートにつきコードブロックは1〜2個まで
- 説明テキストは3〜5文まで
- 読むのに1分以内で済む量

**ユーザーの反応に応じた対応:**
- 質問があれば答える
- 「わかった」「OK」等 → 次のパートへ進む（最後のパートなら手順3へ）
- 「もっと詳しく」等 → 同じパートを掘り下げる

**パート1の出力フォーマット例:**

```
## 📖 レッスン: for...of で配列を走査（1/3）

### for...of とは？
for...of は配列の各要素を順番に取り出すシンプルな構文です。
従来の for ループと違い、インデックス変数を管理する必要がありません。

### 基本構文

\```javascript
for (const item of array) {
  // item に各要素が順番に入る
}
\```

### 例: 配列の要素を出力

\```javascript
const fruits = ["apple", "banana", "cherry"];
for (const fruit of fruits) {
  console.log(fruit);
}
// => "apple"
// => "banana"
// => "cherry"
\```

---
ここまで大丈夫ですか？
```

**パート2の出力フォーマット例:**

```
## 📖 レッスン: for...of で配列を走査（2/3）

### 例: 合計を求める

\```javascript
const nums = [10, 20, 30];
let total = 0;
for (const n of nums) {
  total += n;
}
console.log(total); // => 60
\```

変数を用意して、ループの中で更新していくパターンはよく使います。

---
ここまで大丈夫ですか？
```

**最終パートの出力フォーマット例:**

```
## 📖 レッスン: for...of で配列を走査（3/3）

### for ループとの使い分け

\```javascript
// for ループ: インデックスが必要なとき
for (let i = 0; i < arr.length; i++) {
  console.log(i, arr[i]);
}

// for...of: 要素だけ必要なとき（こっちがシンプル）
for (const item of arr) {
  console.log(item);
}
\```

---
質問はありますか？OKなら練習問題に進みましょう！
```

#### 手順 2-R: 間隔反復復習の出題

復習対象ステップの元ファイルを読み、そのステップで学んだ概念を把握する。
**元の問題と同じ問題は出さず**、同じ概念を使う**新しい問題**を1問出題する。

ファイル名: `review-sr-{ステップID}-r{復習回数}.js`（例: 2回目の復習 → `review-sr-1-3-r2.js`）
初回復習（`reviews` が空の状態）は `r1`、以降インクリメントする。
配置先: 元のステップと同じディレクトリ
過去の復習ファイルは上書きせず、回ごとに別ファイルとして残す。

テンプレート:

```javascript
// =============================================
// 間隔反復復習: Step {ID}「{テーマ}」
// 完了日: {completedAt} / 復習回数: {reviews.length}回目
// =============================================

// 元のステップで学んだ概念:
// - {概念の要約}

// =============================================
// 復習問題: {新しい問題の説明}
// =============================================

function solution(/* 引数 */) {
    // TODO: ここに実装を書いてください
}

// --- テスト ---
console.assert(solution(/* テスト入力1 */) === /* 期待値1 */, "テスト1 失敗");
console.assert(solution(/* テスト入力2 */) === /* 期待値2 */, "テスト2 失敗");
console.assert(solution(/* テスト入力3 */) === /* 期待値3 */, "テスト3 失敗");

console.log("全テスト完了！エラーが表示されなければ正解です");
```

### 3. ディレクトリの準備

対応するステップのディレクトリが存在しない場合は作成する:
- Step 1 → `src/step01-array/`
- Step 2 → `src/step02-set-map/`
- Step 3 → `src/step03-string/`
- Step 4 → `src/step04-loops/`
- Step 5 → `src/step05-two-pointers/`
- Step 6 → `src/step06-hashmap/`
- Step 7 → `src/step07-easy-basic/`
- Step 8 → `src/step08-easy-advanced/`

`package.json` が存在しない場合は `npm init -y` を実行する。

### 4. テンプレートファイルの生成

plan.md のステップ情報（ID、テーマ、内容、ファイル名）に基づいて、以下の構成でテンプレートファイルを生成する。

**重要: 1ファイル = 1問題の原則**

1つのステップで複数の概念を扱う場合でも、**1ファイルにつき1つの問題（1つのTODO）** にする。
ユーザーが1問ずつ実装→実行→確認できるようにするため。

例: ステップで3つの概念を学ぶ場合 → 説明セクションで全概念を説明した後、問題は1つだけ出題する。
その1問で学んだ概念を総合的に使えるような問題にする。

**概念学習ステップのテンプレート（レッスン後）:**

レッスンフェーズで概念の説明は済んでいるので、テンプレートには説明を入れない。
問題とテストだけのシンプルな構成にする。

```javascript
// =============================================
// Step {ID}: {テーマ}
// =============================================

// =============================================
// 問題: {問題の説明}
// =============================================

function solution(/* 引数 */) {
    // TODO: ここに実装を書いてください
}

// --- テスト ---
console.assert(solution(/* テスト入力1 */) === /* 期待値1 */, "テスト1 失敗");
console.assert(solution(/* テスト入力2 */) === /* 期待値2 */, "テスト2 失敗");
console.assert(solution(/* テスト入力3 */) === /* 期待値3 */, "テスト3 失敗");

console.log("全テスト完了！エラーが表示されなければ正解です");
```

**練習ステップのテンプレート:**

練習ステップはレッスンなしで直接問題を出す。
ただし、問題に必要な最低限のコンテキスト（何を使って解くか等）は記載する。

```javascript
// =============================================
// Step {ID}: {テーマ}
// =============================================

// =============================================
// 問題: {問題の説明}
// =============================================

function solution(/* 引数 */) {
    // TODO: ここに実装を書いてください
}

// --- テスト ---
console.assert(solution(/* テスト入力1 */) === /* 期待値1 */, "テスト1 失敗");
console.assert(solution(/* テスト入力2 */) === /* 期待値2 */, "テスト2 失敗");
console.assert(solution(/* テスト入力3 */) === /* 期待値3 */, "テスト3 失敗");

console.log("全テスト完了！エラーが表示されなければ正解です");
```

**復習ステップのテンプレート:**
- 復習対象のステップ範囲から混合問題を出題
- 過去のステップで学んだパターンを組み合わせた問題にする
- 問題ごとにどのパターンを使うかのヒントは入れない（自分で判断させる）

**LeetCode 計画ステップ（a付き）のテンプレート:**
- LeetCode問題文の要約
- 入出力例
- 制約条件
- `// TODO: どのパターンが使えるか考えてみよう` セクション
- `// TODO: アプローチをコメントで書いてみよう` セクション

**LeetCode 実装ステップ（b付き）のテンプレート:**
- 関数シグネチャ
- テストケース（LeetCodeの例 + エッジケース）
- `// TODO: ここに実装を書いてください`

### 5. ユーザーへの案内

**レッスンフェーズの場合（手順2-Lで概念説明をした直後）:**
- ファイルは生成しない
- 「ここまでの内容は大丈夫ですか？質問があれば聞いてください。OKなら練習問題に進みましょう！」と案内する

**練習テンプレートを生成した場合:**
1. テンプレートファイルのパス
2. 実行コマンド: `node --watch src/stepXX/{filename}`（ファイル保存のたびに自動で再実行される）
3. 「TODO部分を実装したら教えてください。レビューします！」

### 6. レビュー（ユーザーが実装後に再度このスキルを呼んだ場合）

ユーザーが「できた」「レビューして」等と言った場合:

1. 該当ファイルを読む
2. `node` で実行してテストが通るか確認
3. テスト結果に応じてフィードバック:
   - **テスト失敗** → 失敗した旨と実行ログを表示する（ヒントはユーザーが求めるまで出さない）
   - **全テスト通過** → **ディスカッションフェーズ**（手順6-D）へ

#### 手順 6-D: ディスカッションフェーズ

テストが通った後、すぐに完了にせず、コードについてディスカッションする。
以下の観点でユーザーの実装をレビューし、対話を促す:

1. **コードの改善点**（あれば）
   - より読みやすい書き方、慣習的な書き方
   - 細かいスタイルの統一（セミコロン、命名など）
   - パフォーマンスの観点（この段階では深入りしないが、明らかなものがあれば）

2. **別解の紹介**
   - 同じ問題を別のアプローチで解く方法を提示する
   - なぜそのアプローチが有効か、どんな場面で使い分けるか説明する
   - ただし、まだ学んでいない概念（後のステップで扱うもの）を使う場合は「これは後で学びます」と軽く触れる程度にする

3. **発展的な問い**（任意）
   - 「もし条件が○○に変わったら、どう書き変える？」のような問いかけ
   - ユーザーの理解を深めるための軽い思考実験

**フォーマット例:**

```
✅ 全テスト通過！

### コードレビュー
（改善点やスタイルの指摘があればここに。なければ「シンプルで読みやすい良い実装です」等）

### 別解
（別のアプローチを1つ紹介）

---
質問や気になることはありますか？
OKなら次のステップに進みましょう！
```

4. ユーザーが質問したら答える。ディスカッションを続ける。
5. ユーザーが「OK」「次へ」等と言ったら、ステップを完了にする:
   - `plan.md` のチェックボックスを `- [x]` に更新
   - **progress.json を更新する**（→ 手順7へ）
   - 「次のステップに進むには `/leetcode-next` と入力してください」と案内する

### 7. progress.json の更新

#### 通常ステップ完了時

テスト全通過後、`docs/progress.json` に以下を記録する:

```json
{
  "ステップID": {
    "completedAt": "YYYY-MM-DD（本日の日付）",
    "reviews": [],
    "intervalLevel": 0,
    "nextReview": "YYYY-MM-DD（明日の日付）"
  }
}
```

#### 間隔反復復習の完了時

**テスト全通過（成功）の場合:**

1. `reviews` 配列に `{ "date": "本日", "passed": true }` を追加
2. `intervalLevel` を +1
3. 新しい `intervalLevel` に応じて `nextReview` を計算:
   - Level 1 → 本日 + 3日
   - Level 2 → 本日 + 7日
   - Level 3 → 本日 + 14日
   - Level 4 → 本日 + 30日
   - Level 5 → `null`（卒業）
4. 復習用ファイル（`review-sr-*.js`）は削除せず残す

**テスト失敗（不合格）の場合:**

1. `reviews` 配列に `{ "date": "本日", "passed": false }` を追加
2. `intervalLevel` を `max(0, current - 1)` に戻す
3. 新しい `intervalLevel` に応じて `nextReview` を再計算
4. ヒントを出して再挑戦を促す

#### plan.md の🔁復習ステップ完了時

通常のplan.md復習ステップ（🔁マーク付き）は、間隔反復とは別の仕組み。
チェックボックスの更新のみ行い、progress.json には記録しない。

### 8. 復習状況のサマリー表示

進捗表示時に、復習状況もあわせて表示する:

```
📊 進捗: 10/82 完了
[██████░░░░░░░░░░░░░░] 12%
🧠 復習状況:
  - 卒業済み: 1-1, 1-2, 1-3（3ステップ）
  - 次の復習: 1-4 → 3月12日, 1-5 → 3月15日
  - 復習期日超過: なし
📅 Phase1-2完了予測: 2026年6月5日（残り72ステップ / 1.0ステップ/日ペース）
🎓 Phase1-2卒業予測: 2026年7月30日（完了+復習卒業55日）
🏆 Medium安定予測: 2026年12月22日（全~220ステップ+復習卒業55日）
👉 次のステップ: 1-11 ...
```

### 9. 予測日の表示

進捗サマリーに3つの予測を表示する。

#### A. Phase1-2完了予測（📅）

全82ステップを解き終わる予測日。

**計算ロジック:**

1. `progress.json` の全ステップの `completedAt` から、最も古い日付（学習開始日）を取得する
2. 完了ステップ数を数える（progress.json のキー数）
3. 学習開始日から本日までの経過日数を計算する（開始日を含むので `本日 - 開始日 + 1`）
4. ペース = 完了ステップ数 ÷ 経過日数（ステップ/日）
5. 残りステップ数 = 82 - 完了ステップ数
6. 残り日数 = 残りステップ数 ÷ ペース（小数点以下切り上げ）
7. Phase1-2完了予測日 = 本日 + 残り日数

#### B. Phase1-2卒業予測（🎓）

全82ステップの間隔反復が卒業（intervalLevel 5）に到達する予測日。

**計算ロジック:**

最後に完了するステップが卒業するまでに、intervalLevel 0→5 の全復習間隔を経る必要がある:
- Level 0→1: 1日後
- Level 1→2: 3日後
- Level 2→3: 7日後
- Level 3→4: 14日後
- Level 4→5: 30日後
- 合計: **55日**

Phase1-2卒業予測日 = Phase1-2完了予測日 + 55日

#### C. Medium安定予測（🏆）

LeetCode Medium を安定して解けるようになる予測日（Phase 1〜5 全完了 + 復習卒業）。

**計算ロジック:**

`plan.md` の全体ロードマップより「ゼロ → Medium が安定して解ける」= ~200〜240ステップ。中央値の **220ステップ** を推定総数として使う。

1. 残りステップ数 = 220 - 完了ステップ数
2. 残り日数 = 残りステップ数 ÷ ペース（小数点以下切り上げ）
3. Medium安定予測日 = 本日 + 残り日数 + 55日（最終ステップの復習卒業）

#### 表示フォーマット

```
📅 Phase1-2完了予測: YYYY年M月D日（残りNステップ / X.Xステップ/日ペース）
🎓 Phase1-2卒業予測: YYYY年M月D日（完了+復習卒業55日）
🏆 Medium安定予測: YYYY年M月D日（全~220ステップ+復習卒業55日）
```

- ペースは小数第1位まで表示する
- 完了ステップが0の場合（初回起動時）は予測を表示しない
- 学習開始日と本日が同日の場合は、その日のペースで計算する（経過日数 = 1）
